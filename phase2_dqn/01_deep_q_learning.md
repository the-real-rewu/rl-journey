# 2.1 — Deep Q-Learning

Phase 1 solved GridWorld: 16 states, a Q-table with 64 entries, everything computable
by hand. Now consider Breakout: the observation is a 210×160×3 pixel image. The
number of distinct states is 256^(100,800) — a number that dwarfs the atoms in the
observable universe. There is no table.

This section is about the one idea that makes RL scale: **replace the Q-table with
a neural network.**

---

## 1. Function Approximation

Instead of storing Q(s, a) for every (s, a) pair, we learn parameters θ such that:

```
Q(s, a; θ) ≈ Q*(s, a)
```

The network takes a state (a stack of frames) and outputs one Q-value per action.
A gradient update on one observed transition now generalizes — nudging Q(s, a; θ)
also changes Q for visually similar states that share network weights.

This generalization is the whole point. It's also the source of new instabilities.

---

## 2. The Deadly Triad

Naive Q-learning with a neural network diverges. The reason is three properties
that interact badly when combined:

**1. Function approximation** — updating Q(s,a;θ) changes Q for all states, not
just s. Neighboring states in the network's feature space all shift together.

**2. Bootstrapping** — the TD target `r + γ max_{a'} Q(s', a'; θ)` uses the same
network being updated. When you improve Q(s,a;θ), the target moves too. You're
chasing a moving target.

**3. Off-policy learning** — the data comes from a replay buffer filled by an older
policy. The distribution of (s, a) pairs in the buffer doesn't match the current
policy's distribution. The network sees a biased sample of experience.

Each property alone is manageable. Together, they create feedback loops that amplify
errors rather than correct them. DQN's two core innovations each sever one link in
this feedback loop.

---

## 3. Fix 1 — Experience Replay

During play, every transition (s, a, r, s', done) is stored in a **replay buffer**
of fixed capacity N. At each training step, a random mini-batch is sampled from
the buffer instead of using the most recent transitions.

**What it fixes:**

- *Correlation:* consecutive frames are nearly identical (the ball moves 2 pixels).
  Training on consecutive transitions means every gradient update reinforces the
  same pattern. Random sampling mixes transitions from across the entire game history,
  making each batch representative of the full state distribution.

- *Data efficiency:* each transition can be sampled and learned from many times,
  not just once.

**What it requires:** off-policy learning. The transitions in the buffer were
generated by older versions of the policy. Q-Learning is off-policy (its target
uses max, not the behavior policy's action), so this is fine. SARSA would not work
with a replay buffer.

---

## 4. Fix 2 — Target Network

A second copy of the network, **θ⁻**, is used to compute TD targets:

```
TD target = r + γ max_{a'} Q(s', a'; θ⁻)
```

`θ⁻` is frozen for N_target steps (e.g. every 1000 steps), then hard-copied from
θ. The online network θ is updated every step; the target network θ⁻ is updated
rarely.

**What it fixes:** the moving-target problem. Because θ⁻ is frozen, the target
`r + γ max Q(s'; θ⁻)` stays constant for many gradient steps. You're fitting to a
stable goal rather than chasing something that changes every update.

**The analogy:** imagine trying to hit a bullseye when the bullseye moves every time
you throw. Freezing the target is just standing still.

---

## 5. Frame Preprocessing

Raw Atari frames are 210×160×3 uint8 — colorful, large, and contain irrelevant
information. DQN preprocesses them:

| Step | Input | Output | Why |
|------|-------|--------|-----|
| Grayscale | 210×160×3 | 210×160×1 | Color irrelevant; reduces input size |
| Resize | 210×160 | 84×84 | Standard DQN input size; removes score region |
| Normalize | uint8 [0,255] | float32 [0,1] | Stable gradient magnitudes |
| Frame stack | 84×84, 1 frame | 84×84×4 | Encodes velocity and trajectory |

**Why frame stacking?** A single frame is a static snapshot. The ball in frame t
could be moving left or right — you can't tell. Stacking 4 consecutive frames gives
the network a short video clip. Ball position across frames encodes velocity;
velocity determines the correct action.

---

## 6. The Full DQN Loop

```
Initialize Q(s,a;θ) and Q(s,a;θ⁻) = Q(s,a;θ)
Initialize replay buffer B with capacity N

for each episode:
    obs = env.reset()
    state = preprocess(obs)          # grayscale + resize + stack

    for each step:
        action = ε-greedy(Q(state; θ))
        next_obs, reward, done = env.step(action)
        next_state = preprocess(next_obs)

        B.push(state, action, reward, next_state, done)

        if len(B) >= batch_size:
            batch = B.sample(batch_size)
            targets = r + γ * max_{a'} Q(s'; θ⁻) * (1 - done)
            loss = MSE(Q(s, a; θ), targets)
            θ ← θ - α ∇_θ loss

        if step % N_target == 0:
            θ⁻ ← θ             # sync target network

        state = next_state
        if done: break
```

The `(1 - done)` term zeros out the bootstrap when the episode ends — same
convention as Phase 1.

---

## 7. The CNN Architecture

Input: 4 × 84 × 84 (four stacked grayscale frames).

```
Conv(32, 8×8, stride 4) → ReLU    # 32 filters capture large-scale structure
Conv(64, 4×4, stride 2) → ReLU    # 64 filters refine spatial features
Conv(64, 3×3, stride 1) → ReLU    # 64 filters capture fine structure
Flatten
Linear(3136 → 512) → ReLU
Linear(512 → num_actions)          # one Q-value per action, no activation
```

This is the exact architecture from Mnih et al. (2015). The output has no
activation — Q-values are unbounded real numbers.

### Exercise 1 — Why a Q-table is impossible

How many Q-values does the table store? How many bytes at float32?

**Part B:** A Breakout frame is 210×160×3 uint8. Treating each unique pixel

### Exercise 2 — The moving target problem

You have a DQN with **no target network** (θ⁻ = θ, always). Current Q-values:
Q(s, a) = 5.0, Q(s', best) = 3.0. Observe transition: r = 1, γ = 0.9, α = 0.1.

Because θ is shared, updating Q(s,a) also changes Q(s', best) slightly.
Say the update shifts Q(s', best) from 3.0 to 2.8 as a side effect of the
gradient step.

**Part A:** Compute the TD target and the updated Q(s, a) after one step.

**Part B:** On the *next* step you observe the same transition again.
Recompute the TD target using the new Q(s', best) = 2.8. Is the new target
higher or lower than before? What direction is Q(s, a) now being pushed?

**Part C:** Sketch what happens over many steps. Does Q(s, a) converge toward
the true value, or does it drift? Why?

**Part D:** With a frozen θ⁻: Q(s', best; θ⁻) stays at 3.0 for 1000 steps.
How does this change your answer to Part C?

### Exercise 3 — What the replay buffer actually does

An agent plays Breakout for 200 steps. For the first 150 steps it explores
randomly and fills the buffer with varied experience. Then it gets stuck near
the left wall for steps 151–200: all 50 transitions look nearly identical
(paddle at x=5, ball at various y positions).

The replay buffer capacity is N=1000, batch size is 32.

**Part A:** You sample a random mini-batch of 32 from the buffer
(which has 200 transitions). In expectation, how many of the 32 come from
the stuck period (steps 151–200)?

**Part B:** If instead you train on the most recent 32 steps (no buffer),
how many come from the stuck period?

**Part C:** The gradient computed from the batch updates weights for states
similar to those in the batch. What does the agent risk learning if all 32
samples are from the stuck period? What do diverse samples protect against?

### Exercise 4 — Why you need 4 frames, not 1

Breakout ball is at pixel position x=80 in the current frame.

**Scenario A:** The ball was at x=78 two frames ago and x=79 last frame.
**Scenario B:** The ball was at x=82 two frames ago and x=81 last frame.

**Part A:** Can a network using only the current frame distinguish A from B?
What information is missing?

**Part B:** What action should the agent take in each scenario?
(Breakout actions: NOOP, FIRE, RIGHT, LEFT — paddle must intercept the ball.)

**Part C:** How does stacking the last 4 frames solve this? What does the
network learn to extract from the stack that it can't extract from one frame?
